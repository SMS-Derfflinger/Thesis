\chapter{文件系统模块的 Rust 重构}

\section{重构需求与总体设计}

原系统文件系统延续 Unix V6 的基本设计，以超级块、磁盘 inode、内存 inode、目录项和数据块为核心数据结构，并通过系统打开文件表和进程文件描述符表组织文件访问过程。其功能包括文件创建与删除、路径解析、目录遍历、文件读写、块映射和空间分配，能够覆盖教学中需要观察的 Unix 文件系统主路径。

原实现主要基于 C++ 对象和裸指针组织模块关系，内存 inode、文件对象和打开文件表之间的生命周期依赖调用者维护。路径解析、引用计数和资源回收逻辑分散在多个模块中，所有权边界不够明确。重构的目标是在不改变文件系统结构和算法思想的前提下，重新组织内部表示和资源管理路径。

Rust 重构保留超级块管理空闲块与空闲 inode、内存 inode 表缓存磁盘 inode、文件对象记录打开状态、目录项支持路径查找等机制，使新旧实现能够在教学逻辑上对应。变化主要集中在数据表示、对象引用关系和错误处理方式上。

实现层面，重构使用 Rust 结构体和枚举表示超级块、inode、文件对象和目录项等核心数据结构；使用 \texttt{Arc<Spin<\_>>}、包装引用类型和 Drop 机制管理内存 inode 与文件对象的生命周期；使用 \texttt{Result<T, E>} 和 PosixError 组织内部错误传播与系统调用返回值映射。这样处理后，文件系统的核心功能保持不变，但资源关系和错误路径更容易审查。

Rust 版本文件系统主要由 src/fs.rs 及其下的 file\_system.rs、inode.rs、file.rs、open\_file\_manager.rs 和 file\_manager.rs 组成。其中，file\_system.rs 负责超级块和磁盘空间管理，inode.rs 负责 inode 对象及文件块访问逻辑，file.rs 负责文件对象和进程文件描述符表，open\_file\_manager.rs 负责系统打开文件表与内存 inode 表，file\_manager.rs负责路径解析和面向系统调用的文件操作流程。

在模块关系上，FileManager 位于上层，负责组织 open、creat、link、unlink、stat 等系统调用对应的文件操作；InodeTable 和 OpenFileTable 位于中间层，分别管理内存 inode 和系统文件对象；FileSystem 负责超级块、空闲块和空闲 inode；底层块访问通过缓冲区缓存和块设备接口完成。该分层使文件系统与设备管理、进程管理和系统调用接口之间的依赖更集中。

\section{文件系统核心数据结构重构}

在超级块的数据结构与状态管理上，Rust 版本将超级块划分为磁盘表示和内存表示两层。磁盘上的超级块使用DiskSuperBlock 结构表示，用于对应磁盘上的固定布局；内存中的超级块使用SuperBlock 结构表示，在保留空闲块表、空闲 inode 表和时间信息的基础上，增加了只读状态、修改状态以及用于同步的锁标志封装。这样做既保留了与原磁盘格式的兼容性，又使内存中的控制信息更加明确。

在状态管理方面，Rust 版本将空闲块表锁和空闲 inode 表锁封装为 SleepFlag，并通过方法接口统一管理锁定、解锁和等待通道。超级块的装载、修改和写回逻辑集中在 load\_super\_block、write\_super\_block 和 update 等函数中，减少了其他模块对内部字段的直接依赖。

在 Rust 版本中，内存 inode 仍然是文件系统运行的核心对象。Inode 结构保留了原系统中的主要字段，包括状态标志、文件模式、引用计数、设备号、inode 号、文件大小和块地址表等，同时使用 bitflags 定义 InodeFlag 和 InodeMode，使标志位操作具有更明确的语义。与原系统相比，这种方式使 inode 状态判断、权限检查和文件类型识别更加直观。

生命周期管理是文件系统重构的重点。系统不再直接传递裸指针，而是引入 InodeSlot、InodeRef 和 InodeRefGuard 等包装类型。InodeRef 在克隆时增加引用计数，在析构时减少引用计数，形成 RAII 风格的自动管理；InodeRefGuard 则限制临时借用场景中的释放时机，避免路径解析或表项操作中出现引用关系混乱。配合 InodeTable 中的 i\_get、i\_dup 和 i\_put\_slot，内存 inode 从装载、共享到回收的过程被组织为较明确的生命周期路径。

打开文件表与文件描述符表也采用类似处理。File 结构保留打开标志、引用计数、关联 inode 和读写偏移量等信息，但管理方式不再依赖全局裸指针数组，而是以 \texttt{Arc<Spin<File>>} 为基础封装为 FileSlot 和 FileRef。FileRef 在克隆时增加引用计数，在析构时调用回收逻辑，从而减少显式关闭和手工维护计数带来的错误风险。

系统打开文件表由 OpenFileTable 管理，负责在系统范围内分配空闲文件对象；进程级文件描述符表由 OpenFiles 管理，负责完成文件描述符到文件对象的映射。文件打开时，系统先在进程文件描述符表中分配可用槽位，再在系统打开文件表中找到空闲文件对象，并建立两者之间的关联。关闭文件时，系统根据引用计数决定是否真正回收底层资源。这种分层管理方式与原系统逻辑一致，但通过引用包装和自动析构机制，使对象关系更加稳定。

\section{文件系统核心功能实现}

目录项在 Rust 版本中由 DirectoryEntry 结构表示，仍采用固定长度目录名和 inode号组成的经典格式。路径解析由 FileManager::find 及相关搜索逻辑完成。其基本思路与原系统一致：从根目录或当前工作目录出发，逐级解析路径分量，在每一级目录中查找对应目录项，并根据查找模式决定是继续向下、创建新项还是删除旧项。

实现上，Rust 版本将目录遍历、权限检查和目录项搜索逻辑封装在 InodeRefExt trait 及其 search 方法中。路径解析不再完全依赖分散的过程式函数，而是转化为围绕 inode 对象的操作接口。目录项搜索过程中同步更新用户态参数区中的目录项信息和偏移量状态，为后续创建、删除等操作提供上下文。该实现保留 Unix 目录查找过程，同时使调用层次更集中。

文件读写仍然通过文件对象、inode 和缓冲区缓存共同完成。对于普通文件，FileManager 根据文件描述符找到对应 FileRef，检查读写权限后，根据当前文件偏移量调用 inode 的读写接口；inode 再根据逻辑块号定位到对应物理块，通过缓冲区缓存读入或写出数据，并在必要时更新文件大小、访问标志和修改标志。

在具体实现中，读操作会根据文件偏移量和剩余长度逐块处理数据，并支持顺序访问时的预读机制；写操作则会在需要时为文件分配新块，并根据写入是否跨越整块决定采用同步读后写、延迟写或异步写的方式。字符设备和块设备文件也被纳入同一读写框架，通过文件类型区分具体处理路径。这样，Rust 版本在保持原有读写模型的同时，使文件对象、inode 和块缓存之间的协作更加明确。

块映射逻辑主要集中在 Inode::get\_blk 及相关辅助函数中。当前实现延续了原系统的多级块索引结构，支持小文件的直接块访问、大文件的一次间接块访问以及更大文件的二次间接块访问。对于尚未分配的块，系统会通过文件系统空闲块分配接口申请新块，并在必要时更新 inode 中的地址表和修改标志。

该设计保留 Unix V6 风格文件系统的块索引方式。Rust 版本将直接块、一次间接块和二次间接块的处理逻辑分层组织，并借助显式缓冲区对象和类型化块号减少误用。块映射算法没有发生根本变化，但代码结构更适合分析和维护。

文件系统的上层功能最终需要通过系统调用接口暴露给用户程序。Rust 版本中，src/fs.rs 集中导出了 open、close、creat、link、unlink、chdir、stat、seek、dup、pipe等系统调用入口，而 interrupt/system\_call.rs 则负责在陷入内核后将系统调用号分发到相应的文件系统处理函数。

这种实现方式使文件系统模块与中断系统、用户态运行时之间形成较清楚的边界。系统调用入口负责参数转发和错误回传，具体逻辑仍由 FileManager、OpenFileTable、InodeTable 和 FileSystem 等内部模块承担。

\section{错误处理与安全性分析}

Rust 版本文件系统广泛使用 Result<T, E>组织内部函数返回值。无论是超级块装载、inode 获取、路径查找、块分配还是文件读写，大多通过 Ok 和 Err 显式表达执行结果。与原系统中常见的通过全局状态或隐式返回值传递错误的方式相比，这种写法使错误传播路径更清晰，函数调用者必须显式处理失败情况。

这种改造对文件系统尤其重要，因为文件系统内部操作链条较长，从路径解析到inode 装载，再到块读取和设备访问，任一步骤都有可能失败。使用 Result 后，错误可以沿调用链逐级上传，并在更高层统一转换为系统调用可识别的错误码，减少了中间层漏处理错误的风险。

同时，尽管内部大量采用 Rust 风格的错误返回方式，但模块对外仍需维持 Unix兼容的错误语义。因此，Rust 版本保留了 PosixError 枚举，用于表示 ENOENT、EACCES、ENOSPC、EBADF、EROFS 等常见错误。文件系统内部的专用错误类型，如 FileSystemError 和缓冲区错误，在需要对外暴露时会进一步映射为 PosixError，再通过系统调用入口写回用户态。

这种分层错误处理方式使内部错误可以保持较细粒度，而对外仍维持传统 Unix 程序所期望的返回语义。文件系统内部不必完全受限于 POSIX 错误码，系统调用边界处再统一映射为用户态可见错误。

从资源管理角度看，Rust 重构的主要变化体现在引用关系和对象生命周期上。原系统中的内存 inode、文件对象和文件描述符表存在跨模块共享关系，计数和释放时机依赖手工维护。Rust 版本通过 \texttt{Arc<Spin<\_>>}、包装引用类型和 Drop 回收机制，将这些关系纳入较明确的生命周期模型，从而降低资源泄漏和错误释放的概率。

从维护角度看，重构后的超级块、inode、文件对象、打开文件表、路径解析和系统调用接口之间的职责边界更明确，错误传播路径也更容易追踪。底层块访问、同步和设备交互仍然需要较低层实现，但文件系统主体逻辑已被组织为更清晰的层次结构。

\section{本章小结}

本章分析了文件系统模块的 Rust 重构。重构保留超级块管理、inode 缓存、目录项查找、文件对象管理和多级块映射等 Unix V6 风格机制，主要调整数据结构表示、对象生命周期和错误处理方式。超级块、inode、文件对象和打开文件表之间的关系经过重新梳理后，路径解析和文件读写逻辑具有更清楚的接口层次，错误传播和 POSIX 映射也更集中。上述变化在保持教学逻辑的同时，提高了文件系统模块的可维护性。
