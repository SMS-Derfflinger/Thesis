\chapter{内存管理模块的 Rust 重构}

\section{重构需求与总体设计}

原系统的内存管理模块承担了多个方面的职责，包括物理页分配、内核内存分配、用户空间管理、页表映射以及交换区管理等。在 C++版本中，这些功能主要围绕PageManager、KernelPageManager、UserPageManager、MemoryDescriptor 和SwapperManager 等对象展开，整体上延续了传统 Unix 教学系统以内核页池和用户页池为基础、结合分页和交换机制的设计思路。运行流程上，系统启动后首先划定内核页池和用户页池的边界，初始化对应的管理结构。之后，进程创建和执行过程中需要申请用户空间页、建立页表映射，并在内存不足时借助交换区转移部分进程映像。内核自身在运行过程中还需要频繁申请小块动态内存，用于进程控制块、缓冲区对象、文件对象及其他内核数据结构的维护。原系统虽然具备基本的内存管理功能，但其分配逻辑主要围绕“页”为中心展开，对不同粒度内存请求的支持不够分层。原系统的物理内存分配以页为基本单位，内核页池和用户页池分别维护各自的可用内存区域。其核心思路是基于预先划定的内存范围，直接管理空闲页块的起始地址和大小信息。分配时，系统根据请求大小按页对齐，在空闲表中线性查找能够满足需求的连续内存区；释放时，再将对应区间归还到空闲表中。整个过程只有基本功能，算法结构相对直接。这种实现方式的优点是逻辑比较直观，便于说明页式分配的基本思想，也容易和内核地址空间布局结合起来理解。但是分配策略较为粗粒度，缺少对不同大小请求的细分处理；而且对内存碎片的控制能力有限。因此，当系统中出现大量小块动态内存请求时，原实现的效率和空间利用率都不够理想。因此后续决定引入Buddy + Slab 分层分配体系。Rust 重构过程中，内存管理模块主要遵循三个原则。第一，保留原系统的教学逻辑。也就是说，仍然沿用页式管理、内核与用户空间分离、交换区辅助运行等总体思路，避免因过度改造而破坏教学连续性。第二，增强模块的独立性。原实现中一些资源管理逻辑较为分散，Rust 版本希望通过更明确的页对象、地址类型和分配接口来建立更清晰的模块边界。第三，增强安全性。由于内存管理本身不可避免地涉及裸地址、页表和底层硬件资源，完全避免 unsafe 并不现实，因此重构的重点不在于避免所

有不安全代码，而在于将不安全操作压缩在少数边界内，并在更高层通过类型系统和所有权机制提供安全封装。

Rust 版本中，内存管理代码被明确划分为多个相互协作的子模块。page.rs 负责定义物理页对象 PhysPage、页集合 Pages 以及页链表等基础抽象；page\_manager.rs负责基于 Buddy 算法组织页分配器，并分别维护内核页与用户页两个分配实例；allocator.rs 接入全局内存分配器，在内核动态分配层面结合 Slab 与 Buddy 两种机制；zone.rs 负责描述整个物理内存区和页元数据数组；swapper\_manager.rs 负责交换区的空间组织。

\section{物理内存管理机制的重构}

\subsection{物理页数据结构设计}

Rust 版本首先对物理页这一基础的内存管理对象进行了重构。在新实现中，每个物理页统一表示为 PhysPage 结构，其中包含链表链接、页块阶数、是否属于 Buddy 系统、是否用作 Slab 页等元信息。同时页对象内部还预留了联合数据区，用于存储 Slab页状态或用户页附加状态。这样一来，物理页不仅是一个可分配的单位，也是承载不

同内存子系统状态的核心对象。相比原系统的实现方式，这种设计有两个明显改进。首先，页的状态信息被集中

到统一的数据结构中，便于页分配器、Slab 分配器和地址空间管理模块共享使用。其二，页对象本身具备更强的可扩展性，可以根据需要在不同场景下复用，可以用于表示普通页、Slab 页或用户页。对于教学系统而言，这种设计保留了物理页是最基本资源单元的理念，同时体现了现代内核实现中常见的对象化管理思路。

\subsection{地址与页框的类型安全抽象}

在原有 C++实现中，地址和页框大多直接以整数形式参与运算，这种方式虽然直观，但容易在不同语义层次间混用。Rust 版本引入了更明确的地址与页框抽象，例如PAddr 表示物理地址，PFN 表示物理页框号，PRange 表示物理地址范围。这些类型来自底层内存管理支持库，并在页分配器、页表和地址转换函数中被广泛使用。这种类型化抽象的价值主要体现在两个方面。首先，它使内存管理代码的语义更加清晰，例如页框号和物理地址在接口层面不再被视为同一种普通整数。另一方面，它降低了实现中出现地址误用的可能性，使一些原本需要依赖经验保证的约束，提前转化为编译期的类型约束。对于本课题而言，对类型的支持也是 Rust 相比原 C++实现的重要优势之一。

\subsection{基于 Buddy 的页分配器实现}

在物理页分配机制上，Rust 版本引入了 Buddy 分配算法作为核心方案。新实现分别维护 KERNEL\_PAGE\_MANAGER 和 USER\_PAGE\_MANAGER 两个 Buddy 分配器，它们共享同一个物理内存区描述，但分别服务于内核页池和用户页池。初始化阶段，系统首先建立页的元数据区，然后根据原系统的地址划分，将内核页池和用户页池对应的物理范围注册到 Buddy 分配器中。Buddy 分配算法的基本思想是将连续页块按照 2 的幂次进行管理。申请内存时，根据请求大小向上取整，换算出对应的阶数，再从空闲链表中分配合适大小的页块；释放时则按照伙伴关系尝试向上合并，以减小外部碎片。与原系统较为直接的页分配逻辑相比，Buddy 的优势在于结构清晰、合并机制明确，并且天然适合连续页申请场景。尤其是在内核需要为较大对象、页表或栈空间提供整页资源时，Buddy 比简单的

线性或链表式分配更具扩展性。

\section{内核堆与 Slab 分配机制重构}

\subsection{内核动态内存分配需求分析}

操作系统内核中的内存请求并不总是以整页为单位出现。大量内核对象的大小都远小于一个页，例如链表节点、文件对象、页表辅助结构、同步原语或状态描述结构等。如果继续像原系统那样主要依赖页级分配来处理所有动态内存请求，就会造成较大的内部碎片，同时也会降低小对象分配和释放的效率。因此，仅有页分配器还不够，还需要在页之上建立更细粒度的对象分配机制。底层仍然应当按页管理物理资源，但内核堆层面需要一种能够有效复用页内空间、适合高频小对象分配的方案。在这一需求下，Rust 版本引入了 Slab 分配机制，并将其与Buddy 系统结合起来，形成分层式的内存分配结构。

\subsection{Slab 分配器的引入与实现方式}

在当前实现中，内核全局分配器通过 GlobalAlloc 接口接入系统，并根据分配请求大小决定采用哪种策略。当申请大小不超过 2048 字节时，请求交由SLAB\_ALLOCATOR 处理；当申请大小超过这一阈值时，则直接从 Buddy 分配器获取整页或多页块。这样做的目的是把小对象分配和页分配区分开来，使不同粒度的请求都能以更合适的方式处理。Slab 分配器本身并不直接管理底层物理页，而是通过 SlabPageAlloc 接口向Buddy 系统申请新的页作为 Slab 页。当系统需要新的 Slab 页时，先从内核 Buddy 分配器中取出一个 0 阶页，再调用页对象上的初始化逻辑建立空闲槽位结构。随后，Slab分配器在该页内部管理固定大小的小对象槽位，并维护页内已分配对象数和空闲链表指针等状态。这样的实现方式体现了上层对象分配器依赖下层页分配器供页的分层结构。为了支持 Slab 分配器运行，页对象中专门维护了与 Slab 相关的状态，例如空闲槽位链表指针和当前已分配对象计数。在 Rust 版本中，物理页不仅是被动的存储单元，而是主动参与到对象分配过程中的状态载体。当一个页被用作 Slab 页后，它会记录自身的槽位使用情况，供分配器后续快速复用。这种设计相较原系统的主要优势在于，它把对象级分配状态明确地绑定到了页对

象之上，使页和页内对象之间的关系更加清楚。对教学而言，这种实现也更容易说明内核堆为何通常不是直接从底层页分配器逐次申请小块内存，而是通过页内切分的方

式提升效率和空间利用率。

\subsection{Buddy 与 Slab 的协同工作机制}

Buddy 与 Slab 在新系统中并不是相互独立的两套机制，而是上下层协同工作的整体 Buddy 负责管理物理页这一粗粒度资源，是整个内存分配体系的基础；Slab 则建立在 Buddy 提供的页之上，面向小对象实现细粒度分配。对于较大的内存请求，系统直接从 Buddy 获取页块；对于较小的内存请求，则从 Slab 页中分配对象槽位。释放时也按照相同层次返回对应资源。

\section{页表与地址空间管理的重构}

Rust 版本中定义了 PageDirectoryEntry、PageTableEntry、PageDirectory 和PageTable 等结构，并使用 bitflags 对页项中的属性位进行封装，例如存在位、可写位、

用户位和大页位等。这种做法相比原系统中的位域结构和直接位操作，更便于在语言层面组织页表项语义，也使相关代码更易读。

从设计上看，这部分重构主要是把原有 IA32 分页机制的表示方式迁移到了Rust，而不是对地址空间模型进行根本性改变。它的意义在于为系统后续完全脱离C++ 的机器相关实现奠定基础，同时也让页表相关逻辑可以与新的地址类型和页对象抽象更自然地衔接。Rust 版本保留了原系统的基本地址空间布局：内核映射位于高地址空间，用户空间位于低地址区域，并通过固定数量的用户页表建立进程可见的虚拟地址空间。在初始化过程中，系统会设置内核页目录项、建立内核页表映射，并初始化用户页表数组。这部分实现更多是在 Rust 中复现原有 IA32 系统的地址空间组织策略，而非重新设计全新的虚拟内存模型。在进程切换过程中，内核需要能够访问当前进程对应的用户结构和相关映射信息。Rust 版本中，系统通过修改内核页表中特定页表项，使其指向当前进程对应的页框，并在修改后刷新页目录，从而实现对当前进程用户结构映射的切换。这一机制延续了原系统的基本思路，但通过 Rust 提供了更统一的接口实现。

\section{交换管理机制的重构}

原系统通过预留交换区来支持进程映像的换出与换入，以在物理内存不足时维持系统运行。Rust 版本保留了这一总体思想，但在交换区的内部管理方式上进行了重构。当前实现中，交换区由一个空闲区间表来描述，每个区间记录起始位置和大小，系统初始化时将整片交换区作为一个大的空闲区域加入管理结构。这种做法比原系统中较为过程化的交换区维护方式更直观。它将交换区明确视为一种连续空间资源，而不是零散地嵌入在换出换入逻辑中的附属结构。这样不仅提高了代码的清晰度，也为后续讨论分配策略和区间合并逻辑提供了明确的数据基础。在交换区分配时，系统会遍历当前空闲区间表，寻找第一个能够满足请求大小的区间。如果找到的区间大于请求大小，则从区间起始处分割出所需部分，并缩减剩余空闲区间；如果大小刚好匹配，则直接将该区间从空闲表中移除。释放时则按照地址顺序将区间插回，并尝试与前后相邻空闲区间合并，以减少碎片。这一实现在算法上并不复杂，优点在于逻辑明确、可验证性较好，并且与 Rust 的

数据结构表达方式比较契合。相比原系统中更偏传统过程式的写法，新的交换区分配器在结构上更容易维护，也更便于后续替换为更复杂的管理策略。

从功能角度看，Rust 版本并没有改变交换机制的基本用途，仍然是为内存不足时的进程换出换入提供磁盘空间支持。但实现方式上，Rust 版本更强调把交换区当作一种独立资源进行管理，使用专门的数据结构维护空闲区间状态，并将分配、释放和合并逻辑显式化。相比原系统，这种方式在代码组织上更清晰，也更容易解释和扩展。与 Buddy + Slab 相比，交换区重构更多体现为将原有机制用更现代、更清楚的方式重新表达。

\section{本章小结}

本章围绕 Unix V6++内存管理模块的 Rust 重构进行了分析。整体来看，重构工作并未改变原系统以内存页为核心、结合分页和交换机制支撑进程运行的基本思想，而是在此基础上重新组织了页对象、页分配器、内核堆分配器和交换区管理逻辑。与原系统相比，新实现的结构更加分层，接口更加清晰，也更便于扩展和维护。最核心的改进体现在 Buddy + Slab 的协同分配体系上，两者共同构成了新的内核内存分配框架。这种分层设计改善了原系统页分配算法较为简化的问题，也使内核动态内存管理能够同时兼顾效率、空间利用率和实现清晰性。
