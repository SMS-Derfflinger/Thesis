\chapter{原系统问题与总体方案}

\section{原系统结构与核心机制}

Unix V6++ 是同济大学面向操作系统课程开发的教学内核，其设计来源于 1975 年发布的 Unix Version 6。原系统在保留 Unix V6 进程、内存、文件和设备等基本结构的基础上，结合 PC 平台环境使用 C++ 在 IA32 架构上重新实现。与功能完备的通用操作系统相比，Unix V6++ 的代码规模较小，模块边界相对清楚，适合作为源码阅读和课程实验对象。

从结构上看，Unix V6++ 采用分层组织方式。boot、machine 和 interrupt 等模块处理启动、分页、中断、时钟和设备控制等平台相关逻辑；proc、mm、fs 和 dev 等模块分别对应进程管理、内存管理、文件系统和设备管理；上层通过系统调用接口向用户程序提供运行环境。其中，machine 模块维护 GDT、IDT 和页表等 IA32 相关对象，proc 模块实现 fork、exec、wait 和调度逻辑，mm 模块管理页式内存和交换区，fs 模块实现 Unix 风格文件系统，dev 模块通过缓冲区缓存和块设备驱动完成磁盘访问。上述结构保留了传统 Unix 教学系统的基本组织方式，也为后续重构提供了清晰的分析对象。

原系统内存管理模块主要位于 \texttt{src/mm/} 目录，核心类包括 PageManager、KernelPageManager、UserPageManager、KernelAllocator 和 SwapperManager。系统以 4KB 页为基本单位，分别维护内核页池和用户页池。根据 PageManager.h 中的定义，两个页池具有固定的起始地址和大小，内存申请和释放均围绕页框区间展开。

具体实现中，系统通过 MapNode 等结构记录空闲页框区间，并提供页级申请与释放接口。物理内存不足时，SwapperManager 负责在主存和交换区之间移动进程映像，从而维持传统 Unix 风格的换出机制。该实现能够支撑进程运行、用户空间分配和页表映射等基本功能，但页分配策略较为直接，主要面向教学场景下的可用性。由于缺少分层分配和小对象分配机制，内核对象频繁申请时容易产生空间利用率和扩展性问题。此外，内存管理代码与 IA32 分页结构耦合较紧，限制了其在新体系结构上的复用。

原系统文件系统模块主要由 FileSystem、INode、File、FileManager 和 OpenFileManager 等部分组成，相关代码位于 \texttt{src/fs/} 目录。系统延续 Unix V6 的文件系统设计，以超级块、inode、目录项和数据块为核心数据结构，通过块设备组织磁盘上的文件和目录信息。

FileSystem 维护超级块中的空闲数据块表和空闲 inode 表，并通过 IAlloc、IFree、Alloc 和 Free 等接口管理磁盘空间与 inode 资源。文件访问过程由 FileManager 组织路径解析、打开、关闭、创建、链接和删除等操作，OpenFileManager 则维护系统范围内的打开文件表。该模块较完整地呈现了 Unix 风格文件系统的基本路径，但内存 inode、文件对象和打开文件表之间的引用关系主要依赖手工计数和全局对象维护，生命周期边界不够明确。

缓冲区缓存与设备管理机制主要位于 \texttt{src/dev/} 目录，核心对象包括 BufferManager、Buf、DeviceManager、BlockDevice、CharDevice 和 ATADriver。系统采用缓冲区缓存作为文件系统和块设备之间的中间层，用于减少重复磁盘访问，并统一上层文件系统与底层设备之间的数据交换接口。

根据 BufferManager.h 的定义，系统维护固定数量的缓冲区控制块和数据区，并通过 GetBlk、Bread、Bwrite、Brelse 等接口支持同步读写、预读、延迟写和异步写。设备管理部分通过 DeviceManager 组织不同类型设备，块设备访问最终由 ATADriver 完成。该驱动直接操作 IA32 平台的 I/O 端口和 ATA 控制器寄存器，实现扇区级数据传输。由此形成的文件系统、缓冲区缓存和块设备驱动路径具有较好的教学可读性，但也使设备层与 IA32 硬件模型绑定较深。

\section{平台绑定与实现维护问题}

Unix V6++ 原系统建立在 IA32 架构之上，启动过程、分页机制、中断描述符表、任务状态段和设备访问方式均与 x86 32 位平台相关。该设计符合系统最初的教学环境，但随着课程实验逐渐转向 QEMU、RISC-V 和其他现代虚拟化平台，IA32 相关实现成为迁移和维护的主要约束。

IA32 还包含较多历史兼容机制，例如实模式与保护模式切换、分段与分页并存、I/O 端口访问和 PIC 中断控制器。上述机制有助于理解传统 PC 平台，但会在教学中引入额外背景知识。与之相比，RISC-V 的特权级、中断和分页模型更规整，适合作为新的体系结构适配目标。

原系统使用 C++ 的类和封装机制组织内核对象，但底层代码仍大量依赖裸指针、手工资源管理和全局对象。内存页、缓冲区、文件对象和 inode 等资源存在跨模块共享，引用计数和释放时机主要由调用者约定维持。该方式在小规模教学系统中可以工作，但在重构、移植或扩展过程中容易引入悬垂引用、重复释放和状态不一致等问题。

此外，原系统中部分宏、全局对象和隐式依赖使接口边界不够清楚。例如，文件系统和缓冲区缓存之间共享较多底层状态，内存管理和 IA32 页表结构之间也存在较强绑定。随着新增功能增多，调用关系的可追踪性下降，调试和验证成本随之提高。

原系统能够展示经典 Unix 内核机制，但当前操作系统课程还需要覆盖现代指令集架构、虚拟化设备、内存安全和模块化工程实践。原有设备模型主要面向传统 PC 硬件，与 QEMU 环境中常见的 VirtIO 设备存在差异；原有内存管理和文件系统也缺少对资源生命周期的显式表达。

因此，本课题并不否定原系统的教学价值，而是在其既有结构上进行有限重构。Rust 用于改善资源管理和接口表达，RISC-V 用于替换与 IA32 强绑定的平台层。这样的处理方式有助于保持课程内容的连续性，同时使系统更适合后续实验扩展。

\section{总体重构与迁移方案}

本课题在语言层面的目标是使用 Rust 重构原系统中的内存管理和文件系统模块。内存管理部分重点整理页对象、页分配器、内核堆和交换区管理；文件系统部分重点整理超级块、inode、打开文件表、文件描述符表和错误处理路径。重构过程中尽量保留原系统的数据关系和核心算法，以便学生仍能观察到 Unix V6 风格的设计。

Rust 的所有权、借用、智能指针和生命周期机制用于替代部分依赖人工约束的资源管理逻辑。底层页表、地址转换和设备访问仍需要 unsafe 代码，但重构后的实现将这些操作限制在较小边界内，并在上层提供类型化接口。这样可以降低接口误用概率，也使资源申请和释放关系更容易审查。

在体系结构层面，本课题将系统从 IA32 平台迁移到 RISC-V 64 位平台，适配 Sv39 分页、trap 机制、UART 串口和 VirtIO 块设备。迁移目标不是完全重写上层文件系统和进程逻辑，而是重建平台相关支撑，使原有教学结构能够在新的体系结构上运行。
