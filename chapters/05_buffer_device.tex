\chapter{缓冲区缓存与设备管理的 Rust 重构}

\section{重构需求与总体设计}

原系统的缓冲区缓存位于文件系统与块设备之间，是块 I/O 路径中的核心中介层。其主要职责包括缓存磁盘块内容、减少重复 I/O、组织同步读写与延迟写、维护空闲缓冲区队列，并在 I/O 完成后唤醒等待进程。对于 Unix 风格教学系统而言，缓冲区缓存不仅承担性能优化作用，也直接体现了文件系统与设备层之间的耦合方式。从实现角度看，原缓冲区缓存机制依赖双向链表、状态标志和显式的睡眠唤醒逻辑来维护缓冲区生命周期。缓冲区既可能挂在设备缓存链中，也可能位于空闲队列中，还可能进入设备 I/O 请求队列。这样的设计具有典型的 Unix 内核风格，但在 C++实现下，链表指针、状态位和资源回收逻辑高度依赖人工维护，容易出现状态不一致、重复挂接和释放路径复杂的问题。因此，对缓冲区缓存进行 Rust 重构，有助于在保留原始算法结构的同时提升代码组织的清晰度。设备管理方面，原系统采用传统 Unix 设备分层思路，将设备划分为块设备和字符设备两类。块设备主要服务于文件系统和交换区，字符设备则面向终端等顺序访问场景。设备访问路径通常为：上层模块发起请求，设备管理层根据设备号选择对应设备对象，再进入具体设备驱动执行实际 I/O 操作。在教学系统中，这种结构有较高的示范意义，清楚展示了从逻辑设备号、设备分发表到具体驱动实现的映射关系。但是原实现中设备管理和驱动代码与平台细节耦合较深，而且与缓冲区缓存之间存在较多低层共享状态。Rust 重构的目标不是改变这种分层思想，而是在保持设备抽象不变的前提下，用更清楚的数据结构和更明确的接口组织其内部逻辑。Rust 版本在总体结构上保留了原系统的设备分层思想：文件系统和交换区通过缓冲区缓存访问块设备，字符设备通过单独接口提供读写能力，设备号仍然区分主设备号与次设备号。在此基础上，重构工作主要集中在三个方面。首先，对缓冲区对象和缓冲区队列进行重新建模。Rust 版本使用显式的缓冲区结构、状态位封装以及侵入式链表来替代原有的手工链表操作。其次，重构 I/O 请求管理过程，使缓冲区缓存、设备请求队列、中断完成处理和睡眠唤醒之间的关系更加清楚。最后，将块设备层和字符设备层分别抽象为 Rust trait，再由具体驱动实现相应接

口，从而增强设备管理的模块化程度，在保持原系统运行逻辑的前提下，使缓冲区缓存与设备层的实现更适合分析、维护和扩展。

\section{缓冲区缓存机制重构}

\subsection{Buffer 数据结构设计}

Rust 版本中，缓冲区缓存的基础对象包括 Buf、Buffer、BufRef、BufferData 和BufferSlot 等结构。其中，Buf 负责保存缓冲区元数据，包括设备号、物理块号、状态标志、剩余字节数、错误码以及链表链接等；BufferData 保存实际的 512 字节块数据；BufferSlot 则将一个 Buf 与一个数据块绑定在一起，形成系统初始化时预先分配的固定缓冲区槽位。与原系统相比，这种设计将缓冲区的元数据和数据存储分离得更加清晰。Buf 负责状态和队列关系，BufferData 负责内容承载，上层使用时通过 Buffer 这一包装对象访问缓冲区数据。缓冲区在不同队列中流转时仍然保持统一的对象表示，数据区也可以通过方法接口安全访问，这种拆分有利于区分控制信息和数据内容两个层次。

\subsection{缓冲区与队列管理}

缓冲区获取过程由 BufferManager::get\_blk 和内部的 get\_blk\_ref 实现。其基本流程仍然延续了经典 Unix 缓冲区缓存算法：先检查目标块是否已经在缓存中；若已存在且未被占用，则直接取出；若已存在但正忙，则当前进程睡眠等待；若不存在，则从空闲队列中选择一个缓冲区重新分配给目标块。对于带有延迟写标志的缓冲区，系统会先触发异步写回，再重新进行分配。释放过程由 brelse 完成。缓冲区释放时，系统会清除忙标志和等待标志，将缓冲区重新插回空闲队列，并根据需要唤醒等待该缓冲区或等待空闲缓冲区的进程。与原实现相比，Rust 版本将获取和释放逻辑集中封装在 BufferManager 中，并借助更明确的状态检查函数和链表接口减少了手工维护细节的暴露，使缓冲区生命周期更容易追踪。队列管理方面，Rust 版本保留了原系统中空闲队列和设备缓存队列并存的组织方式。空闲队列用于管理当前未被占用的缓冲区，设备缓存队列则按设备组织已经关联到某设备块号的缓冲区。除此之外，Rust 版本还明确引入了设备 I/O 请求队列，用于保存已经提交给块设备但尚未完成传输的请求。在实现上，这些队列由侵入式链表维护。每个 Buf 对象中同时包含设备链、空闲链和 I/O 链三组链接字段，从而支持一个缓冲区在不同逻辑队列中的切换。相比原系

统直接操作多个前驱后继指针的方式，Rust 版本通过侵入式链表适配器统一了队列管理接口，降低了链表操作出错的可能性，同时保留了缓冲区缓存算法本身的结构特点。

为了保留原系统在块 I/O 方面的典型优化策略，Rust 版本继续支持预读、延迟写和异步写。预读由 breada 实现，在顺序访问场景下，系统在读取当前块的同时尝试异步发起下一块的读请求。延迟写由 bdwrite 实现，用于先将数据留在缓冲区中而不立即写盘，从而减少频繁同步写的开销。异步写由 bawrite 实现，用于在后台发起写请求，而不阻塞当前执行流。

\section{同步与中断处理机制重构}

\subsection{睡眠与唤醒机制实现}

缓冲区缓存与设备层的协作依赖于睡眠与唤醒机制。Rust 版本在等待缓冲区可用、等待 I/O 完成和等待交换缓冲区释放等场景中，仍然采用睡眠阻塞当前进程、在条件满足后唤醒继续执行的方式。典型场景包括：目标缓冲区被其他进程占用时，当前进程在该缓冲区等待通道上睡眠；空闲队列为空时，在空闲队列等待通道上睡眠；同步 I/O 发起后，在对应缓冲区通道上等待 I/O 完成。I/O 完成后，由 io\_done 或 brelse 负责唤醒相应等待者。这样设计使块 I/O 路径能够自然地与进程调度结合，不必采用忙轮询方式等待设备完成。Rust 版本在保留原机制的同时，对等待通道和等待条件的表达做了更明确的封装，使睡眠与唤醒路径更容易分析。

\subsection{缓冲区缓存与文件系统协同}

文件系统和缓冲区缓存之间的关系在 Rust 版本中仍然十分紧密。文件系统中的inode 读写、块映射、超级块装载和交换区操作都会通过 global\_buffer\_manager()访问底层块缓存接口。普通文件读写通过 bread、bdwrite 和 bawrite 等接口进行，超级块装载和写回通过缓冲区实现扇区级访问，交换区操作则通过专门的 swap 接口完成。从结构上看，文件系统不直接面对设备驱动，而是统一依赖缓冲区缓存。缓冲区缓存既负责组织块对象的复用，也负责把上层逻辑转换为设备请求。这样做延续了传统 Unix 系统的 I/O 分层方式，同时也使文件系统逻辑可以相对独立于具体块设备实现。Rust 重构保留了这一分层关系，并使模块间接口更加明确。

\section{块设备驱动重构}

\subsection{ATA DMA 驱动实现}

在当前分支中，块设备驱动仍然基于 ATA 控制器和 DMA 机制实现。Rust 版本中的 ATADriver 负责设置 ATA 寄存器、构造 DMA 请求描述符、启动 DMA 传输并在磁盘中断到来时完成请求收尾。驱动代码保留了原系统对 LBA28 寻址、ATA 命令字和 DMA 控制寄存器的使用方式，说明其总体运行模型与原实现保持一致。驱动在发起请求时会根据缓冲区中的设备号、物理块号和传输字节数，填写 ATA控制寄存器和 DMA PRD 表，再启动总线主控 DMA 传输。对于读请求和写请求，驱动分别发送不同的 ATA DMA 命令。

\subsection{块设备请求处理流程}

Rust 版本的块设备请求处理流程可以概括为以下几个阶段。首先，文件系统或交换区通过缓冲区缓存发起读写请求。其次，BufferManager 为目标块分配或获取缓冲区，并调用块设备的 strategy 接口。随后，块设备将请求缓冲区插入设备 I/O 队列；当设备空闲时，start 函数取出队首请求，调用 ATADriver::dev\_start 真正启动传输。传输完成后，磁盘中断入口调用 ATADriver::ata\_handler，检查错误状态、更新设备活动标志、弹出已完成请求，并通知缓冲区管理器执行 io\_done。最后，等待该缓冲区的进程被唤醒，I/O 路径结束。这一流程相比原系统更清楚地分离了三层职责：缓冲区缓存层负责组织块对象与等待机制，块设备层负责管理设备请求队列，驱动层负责实际寄存器和 DMA 操作。

\subsection{设备抽象与分发表组织}

Rust 版本将块设备和字符设备分别抽象为 BlockDevice 和 CharDevice trait。块设备 trait 要求实现 open、close、strategy、start 和 devtab 等接口，字符设备 trait 则提供

open、close、read 和 write 等方法。当前系统中，块设备主要由 ATABlockDevice 实现，字符设备主要由控制台设备实现。设备号解析仍然通过主设备号和次设备号完成，设

备分发逻辑则集中在设备管理辅助函数中。这种做法保留了传统 Unix 设备开关表的思想，但在 Rust 中采用 trait 加静态实例的方式表达，减少了原系统中通过数组和裸指针维护设备对象的复杂性。对论文写作来说，这一点可以作为设备层重构的重要特征，说明设备抽象方式在 Rust 中得到了更清楚的实现。

\section{本章小结}

本章围绕缓冲区缓存与设备管理模块的 Rust 重构进行了分析。重构工作在保留原系统块缓存算法、设备分层结构和 ATA DMA 驱动模型的基础上，重点重构了缓冲区对象表示、队列管理、睡眠唤醒、中断完成路径以及设备抽象接口，块 I/O 路径仍然保持了经典 Unix 教学系统的结构特征，但内部实现更加清晰、分层更明确，这一重构巩固了文件系统与设备层之间的协作关系。
