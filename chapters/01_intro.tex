\chapter{引 言}

\section{课题背景}

操作系统课程通常需要在抽象机制和可观察实现之间建立联系。进程调度、虚拟内存、文件系统和设备 I/O 并不是彼此孤立的概念；它们在一次系统调用、一次缺页处理或一次文件读写过程中同时发生作用。通用教材能够给出较完整的概念框架\supercite{tanenbaum2017modern,you2000unix}，但仅依赖教材讲授时，学生较难理解内核对象之间的生命周期关系、地址空间转换路径以及文件系统与块设备之间的接口边界。教学操作系统以较小规模的代码保留关键机制，使课程实验能够围绕真实内核路径展开。

Unix V6++（Unix V6++ Tongji's Edition for Education）是同济大学陈闳中教授及多届研究生团队开发的教学操作系统。该系统以 Unix Version 6 为基础，在 IA32 架构上使用 C++ 重新实现，覆盖进程管理、内存管理、文件系统、缓冲区缓存和设备管理等模块，代码规模适合课程阅读和实验修改\supercite{chen2013unixv6pp}。与完整工业内核相比，Unix V6++ 没有复杂的硬件兼容层和大规模驱动框架，因而能够较集中地呈现 Unix 风格内核的基本组织方式。

原系统的课程价值并不消除其实现层面的维护压力。IA32 平台包含较多历史兼容机制，启动、分页、中断和传统 ATA 设备接口均带有明显的平台绑定。C++ 实现虽然通过类封装组织了部分内核对象，但文件对象、inode、缓冲区、页框和进程地址空间之间仍大量依赖裸指针、全局状态和手工约定维持引用关系。此类写法便于展示早期 Unix 内核的实现习惯，却使资源释放、错误传播和跨模块共享状态更难审查。对于后续课程实验而言，学生在修改局部代码时容易触发隐藏在对象生命周期和共享状态中的错误。

Rust 和 RISC-V 为教学操作系统的更新提供了较明确的技术方向。Rust 通过所有权、借用检查和 RAII 风格析构机制，将部分资源管理约束前移到类型系统中\supercite{klabnik2022rust}；RustBelt 对 Rust 类型系统及其 unsafe 扩展边界给出了形式化研究，说明 Rust 的安全性依赖于受控的 unsafe 抽象，而不是对底层不安全操作的简单消除\supercite{jung2018rustbelt}。针对 Rust CVE 的经验研究也表明，Rust 项目中的内存安全问题往往集中于 unsafe 代码和不健全抽象边界，因而系统软件使用 Rust 时仍需明确标注和审查 unsafe 范围\supercite{xu2022memorysafety}。这些研究与教学内核重构的目标相吻合，即在保留硬件控制能力的同时，尽可能把裸指针和隐式生命周期约定限制在较小范围。

RISC-V 则提供了较适合教学解释的硬件接口。RISC-V 采用开放指令集规范，其特权级、异常、中断和分页机制具有公开、模块化的描述\supercite{waterman2017riscv,riscv2021privileged}。近年来，RISC-V 已从处理器设计扩展到操作系统、编译器、模拟器和软硬协同优化等系统软件议题。武延军等在《软件学报》RISC-V 系统软件专题前言中指出，RISC-V 技术的发展正在推动操作系统、编译器、测试验证和安全可信等方向的研究\supercite{wu2025riscv}；孙冉等则从人工智能与高性能计算场景综述了 RISC-V 的开放、模块化和可扩展特征\supercite{sun2026riscv}。对于教学操作系统而言，RISC-V 的意义不在于替代所有既有平台，而在于提供一个结构较清楚、文档可获得、便于在 QEMU 中复现实验的目标架构。

基于上述背景，本课题选择对 Unix V6++ 的内存管理、文件系统、缓冲区缓存和设备接口进行 Rust 重构，并在此基础上完成面向 RISC-V 64 位平台的迁移。该工作并非重新设计一个全新的教学内核，而是在原 Unix V6++ 的课程结构上分析 C++ IA32、Rust IA32 和 Rust RISC-V 三个版本的差异，重点讨论资源生命周期、错误处理、模块边界和体系结构适配方式的变化。

\section{相关研究现状}

Unix V6 及其衍生系统长期用于操作系统教学。Lions 对 Unix 第六版源码的注释将早期 Unix 内核的进程、文件和设备结构转化为可阅读的教学材料\supercite{lions1996unix}。xv6 延续这一传统，以较小规模重写 Unix V6，并在后续版本中将教学平台迁移至 RISC-V\supercite{kaashoek2021xv6}。这类系统的共同特点是牺牲工业级完整性，保留教学所需的核心机制。Unix V6++ 与其处于相近谱系，但它具有本地课程积累和 C++ 实现基础，因而更适合作为同济大学操作系统课程的连续实验平台。

Rust 操作系统研究主要沿两条路径展开。一类工作面向教学和实验内核。Writing an OS in Rust 通过逐步构建裸机内核说明了 Rust 在中断、分页和堆分配等场景中的基本用法\supercite{oppermannWritingOS}；rCore-Tutorial 将 Rust 的所有权、生命周期和错误处理纳入操作系统实验过程\supercite{chen2023rcore}；NPUcore 的实践工作则说明了国内课程环境中使用 Rust 构建教学内核的可行性\supercite{zhang2020npucore}。这类系统证明 Rust 能够用于组织页表、分配器和系统调用路径，但也显示出一个共同事实：硬件寄存器访问、上下文切换和页表写入等路径仍需 unsafe 支撑。

另一类工作面向安全内核结构。Tock 在资源受限的微控制器上使用 Rust 构建多程序环境，通过语言安全和有限硬件保护机制组织隔离边界\supercite{levy2017tock}；Levy 等在 APSys 2017 进一步讨论了使用 Rust 编写内核时 DMA、USB 和缓冲区缓存等机制所需的 unsafe 抽象\supercite{levy2017kernelrust}。Theseus 采用 Rust 重新组织操作系统结构和状态管理，试图减少组件之间长期持有对方状态的现象\supercite{boos2020theseus}。RedLeaf 则探索基于 Rust 类型和内存安全的轻量隔离域，并将驱动和系统服务组织为可隔离、可恢复的域\supercite{narayanan2020redleaf}。这些研究表明，Rust 在操作系统中的价值并非简单替换 C/C++ 语法，而是借助类型系统重新划定状态共享、资源所有权和模块边界。

工业界也开始将 Rust 引入内核或内核相关组件。Linux 内核已将 Rust 支持作为实验性基础设施合入主线，用于评估 Rust 在内核抽象和驱动开发中的适用性\supercite{linuxkernelRust}；Asterinas 等项目则尝试以 Rust 重新组织系统调用兼容层、内核对象和驱动接口，为安全内核结构提供了另一类工程样本\supercite{antgroup2023asterinas}。这些实践为教学内核重构提供了现实参照，但工业项目通常包含复杂的兼容目标和工程约束，并不适合直接作为本科操作系统实验对象。

从已有研究看，教学操作系统、Rust 安全内核和 RISC-V 系统软件已经分别形成较多成果，但三者在 Unix V6++ 这一具体系统中的结合仍有研究空间。现有 Rust 教学内核多从零开始构建，较少讨论如何在保留既有课程体系和 Unix V6 风格算法的前提下完成增量重构；RISC-V 教学内核已有 xv6 等代表性系统，但它们无法直接承接 Unix V6++ 的本地课程资产。基于这一空缺，本文以 Unix V6++ 的原始 C++ 版本为参照，在内存管理和文件系统两个资源关系最密集的模块中引入 Rust，并进一步观察该重构对 RISC-V 迁移的支撑作用。

\section{研究意义}

本课题的意义首先体现在教学系统的可维护性改进。内存管理和文件系统是 Unix V6++ 中对象生命周期最复杂的两个模块，涉及页框、页表、交换区、inode、打开文件表、目录项和缓冲区缓存等多类共享对象。Rust 重构可以把原先依赖人工约定维护的引用关系转化为显式类型、引用包装和析构路径，使学生在阅读和修改代码时更容易定位资源归属和错误返回位置。

其次，课题有助于保留 Unix V6++ 的课程连续性。直接替换为全新教学内核虽然可以获得较新的代码结构，但会割裂已有实验、讲义和源码分析材料。本文选择在原系统结构上进行模块化重构，使传统 Unix V6 的超级块、inode、多级块映射、缓冲区缓存、页式内存管理等机制继续保留，同时通过 Rust 重新组织其实现边界。这种方式更适合课程平台的渐进演化。

再次，课题为教学操作系统的体系结构迁移提供了案例。IA32 版本与 x86 分页、中断和 ATA 设备耦合较强，而 RISC-V 版本需要重新处理 Sv39 页表、trap 机制、PLIC 外部中断和 VirtIO MMIO 块设备。通过比较 Rust IA32 和 Rust RISC-V 两个版本，可以观察哪些上层结构能够跨架构复用，哪些部分必须隔离到平台相关模块中。这一过程对理解操作系统的硬件抽象边界具有直接教学价值。

\section{研究目标}

本课题围绕 Unix V6++ 的 Rust 重构和 RISC-V 迁移展开，研究目标可以概括为以下几个方面。

第一，重构内存管理模块。Rust 版本需要保留原系统的页式管理思想和教学用地址空间布局，同时重新组织物理页、页分配器、内核堆和交换区管理。具体工作包括引入类型化地址表示、Buddy 与 Slab 分层分配机制、RAII 风格页资源回收以及更集中的交换区空闲区间管理。

第二，重构文件系统及其下层缓冲区路径。文件系统部分保留 Unix V6 风格的超级块、inode、目录项、打开文件表和多级块映射结构，重点整理内存 inode、文件对象和文件描述符表之间的引用关系。缓冲区缓存和块设备接口需要在保持原有块访问语义的基础上，使用 Rust 类型和 trait 抽象重新组织缓冲区状态、设备队列和 I/O 请求路径。

第三，完成面向 RISC-V 64 位平台的迁移。迁移工作包括 Sv39 页表适配、UART 串口和 VirtIO 块设备驱动等内容。文件系统上层逻辑应尽可能复用，以检验重构后模块边界对跨架构适配的支撑效果。

第四，比较三个版本的功能和性能表现。论文以 main 分支原始 C++ IA32 版本、refactor/x86 分支 Rust IA32 版本和 refactor/riscv 分支 Rust RISC-V 版本为对象，通过功能测试和性能测试分析重构与迁移后的运行表现。评价重点不是追求单项性能最优，而是确认系统能否保持基本教学功能，并在可接受开销内获得更清晰的资源管理和平台适配结构。

\section{论文组织结构}

本文共八章。第 1 章说明课题背景、相关研究、研究意义与研究目标。第 2 章分析原 Unix V6++ 的系统结构，重点梳理内存管理、文件系统、缓冲区缓存和设备接口的实现基础。第 3 章讨论内存管理模块的 Rust 重构，包括物理页抽象、Buddy 分配器、Slab 分配器、交换区管理和地址空间组织。第 4 章讨论文件系统模块的 Rust 重构，重点分析 inode、打开文件表、目录解析和块映射路径。第 5 章说明缓冲区缓存与设备管理的重构方式。第 6 章介绍 RISC-V 架构迁移中的启动、分页、串口和 VirtIO 块设备适配。第 7 章给出功能测试和性能测试结果，并分析三个版本的运行表现。第 8 章总结全文工作，说明现有不足和后续改进方向。
